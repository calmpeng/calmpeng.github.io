### 起因

在 [Blog ](https://www.yinwang.org/blog-cn/2020/02/13/java-type-system) 上看到一个有趣的问题，这段代码里面到底哪一行错了？

```java
public static void f() {
    String[] a = new String[2];
    Object[] b = a;
    a[0] = "hi";
    b[1] = Integer.valueOf(42);
}
```

一开始看到觉得就是第五行代码中，用 Integer 的值赋给了实际上时 String 类型的对象，然后报错出现类型不匹配的问题。

这里其实有几个问题：

1. 为什么编译器和 IDE 在编译过程中没有发现
2. 为什么第三行，数组赋值的时候，向上转型的时候没有提示

了解到这里涉及到 Java 中的**数组协变**，**泛型**的类型转换的一些概念。

**PECS ** 原则：Producer-Extends, Consumer-Super。

---

> 编程语言的设计者在制定**数组、继承、泛型数据类别**等的类型规则时，必须将“变型”列入考量。将类型构造器设计成是协变、逆变而非不变的，可以让更多的程序俱备良好的类型。另一方面，程序员经常觉得逆变是不直观的；如果为了**避免运行时期错误而精确追踪变型，可能导致复杂的类型规则**。为了保持类型**系统简单同时允许有用的编程**，一个编程语言可能把类型构造器视为不变的，即使它被视为可变也是安全的；或是把类型构造器视为协变的，即使这样**可能**会违反类型安全。



> 只读数据类型（源）是协变的；只写数据类型（汇/sink）是逆变的。可读可写类型应是“不变”的。



> **tl;dr:** "PECS" is from the collection's point of view. If you are *only* pulling items from a generic collection, it is a producer and you should use `extends`; if you are *only* stuffing items in, it is a consumer and you should use `super`. If you do both with the same collection, you shouldn't use either `extends` or `super`.



> 总结 ? extends 和 the ? super 通配符的特征，我们可以得出以下结论：
>
> - 如果你想从一个数据类型里获取数据，使用 ? extends 通配符
> - 如果你想把对象写入一个数据结构里，使用 ? super 通配符
> - 如果你既想存，又想取，那就别用通配符。
>
> 这就是Maurice Naftalin在他的[《Java Generics and Collections》](https://link.zhihu.com/?target=http%3A//bit.ly/jAzfIt)这本书中所说的存取原则，以及Joshua Bloch在他的[《Effective Java》](https://link.zhihu.com/?target=http%3A//goo.gl/ixwNn)这本书中所说的PECS法则。
>
> Bloch提醒说，这PECS是指”Producer Extends, Consumer Super”，这个更容易记忆和运用。
>
> [http://www.importnew.com/14985.html](https://link.zhihu.com/?target=http%3A//www.importnew.com/14985.html)
>
> **什么是PECS？**
>
> PECS指“Producer Extends，Consumer Super”。换句话说，如果参数化类型表示一个生产者，就使用<? extends T>；如果它表示一个消费者，就使用<? super T>，
